# 《Ether》全功能开发规划

## 1. 项目目标

本项目最终目标，是在 Godot 4.6 中实现一套参考《The Riftbreaker（银河破裂者）》产品级设计文档的完整玩法系统：机甲动作战斗、基地建设、资源生产、塔防防御、科技研究、关卡推进、生存模式、沙盒模式与多人合作。

实现方向是“功能与系统结构对标”，不是直接复制原游戏的商业素材、商标、剧情文本或专有表现。我们会保留核心玩法逻辑，并发展成自己的世界观、美术资产、数值和关卡内容。

## 2. 当前项目状态

当前项目已有内容：

- Godot 4.6 项目配置。
- 一个主场景 `node_2d.tscn`。
- 一个可移动角色 `CharacterBody2D`。
- 角色前后方向动画。
- 跟随角色的 `Camera2D`。
- 鼠标滚轮缩放相机。
- 一张大地图背景图。
- Godot MCP 插件。

当前缺失内容：

- 没有基地核心。
- 没有资源系统。
- 没有建筑/建造模式。
- 没有敌人、生命值、攻击、波次。
- 没有防御塔自动攻击。
- 没有 UI、任务、胜负流程。
- 没有保存、科技树、关卡管理。

## 3. 总体开发原则

1. 先完成最小可玩闭环，再扩系统。
2. 所有核心数值尽早数据化，不把塔、敌人、资源的数值写死在逻辑里。
3. 基地是所有流程的锚点，敌人、资源、科技、任务都围绕基地推进。
4. 玩家机甲是机动补位，不是唯一输出来源；塔防和基地系统必须成为中后期主轴。
5. 每个阶段都必须有可运行、自测通过的验收标准。
6. 中文文件全部保持 UTF-8 编码。

## 4. 核心玩法闭环

最终闭环：

探索地图 -> 采集资源 -> 扩建基地 -> 建造电力/生产/防御设施 -> 研究科技 -> 抵御敌潮 -> 解锁新区域/新生物群系 -> 获取高级资源 -> 强化机甲与基地 -> 完成战役目标或进入无尽挑战。

第一版闭环：

探索地图 -> 采集单一资源 -> 回基地建造基础防御塔 -> 敌人波次进攻基地 -> 防御塔和玩家共同守住基地 -> 获得奖励并进入下一波。

## 5. 里程碑规划

### 阶段 0：工程基础整理

目标：把当前 Demo 整理成可长期扩展的项目结构。

主要功能：

- 建立目录结构：`scenes/`、`scripts/`、`resources/`、`ui/`、`data/`。
- 拆分玩家、基地、敌人、建筑、资源、UI 的脚本职责。
- 建立全局游戏状态管理器。
- 建立统一信号事件流，例如资源变化、基地受伤、波次开始、波次结束。
- 建立调试 HUD，显示 FPS、玩家坐标、资源、波次、敌人数量。

验收标准：

- 项目能正常打开主场景。
- 玩家移动、动画、相机缩放不退化。
- 新目录结构下资源引用不丢失。

### 阶段 1：最小可玩原型

目标：做出第一条完整的 5 分钟游戏流程。

主要功能：

- 基地核心建筑。
- 基地生命值。
- 单一资源：能量矿。
- 资源点：玩家靠近后按 `E` 采集。
- 基础建造模式：按键切换建塔状态，在基地附近放置哨兵塔。
- 哨兵塔自动寻找敌人并攻击。
- 敌人从固定出生点刷出，朝基地移动并攻击基地。
- 波次系统：准备倒计时、刷怪、清场、结算。
- 简单 HUD：资源、基地血量、波次倒计时、当前波次。
- 胜负条件：基地被毁失败，守住指定波次胜利。

验收标准：

- 玩家能采集资源。
- 资源足够时能建塔，资源不足时不能建塔。
- 敌人会攻击基地。
- 塔会自动攻击敌人。
- 守住和失败都能触发明确结果。

### 阶段 2：战斗与塔防基础扩展

目标：让防御规划开始有策略性。

主要功能：

- 玩家基础攻击。
- 敌人生命值、伤害、移动速度、攻击范围。
- 防御塔射程、攻速、伤害、目标选择。
- 建筑可被敌人攻击并摧毁。
- 城墙系统。
- 修复系统。
- 防御塔升级到 2-3 级。
- 敌人进攻路径优先级：资源点、外部建筑、基地核心。
- 波次敌人数量和强度递增。

验收标准：

- 玩家不建塔会明显难以守住。
- 只靠塔也可能出现漏怪，需要玩家补位。
- 城墙能改变敌人的接近路径或延缓进攻。

### 阶段 3：资源与生产链

目标：把“捡资源”升级成基地运营。

主要功能：

- 多资源：碳、铁、能量、高级矿物。
- 资源存储上限。
- 采集建筑。
- 发电建筑。
- 电力消耗与供给。
- 生产建筑。
- 弹药/特殊资源消耗。
- 资源点枯竭或低产逻辑。
- 资源 UI 明细。

验收标准：

- 建筑运行受资源和电力限制。
- 电力不足会导致部分建筑停摆。
- 玩家需要扩张到新资源点才能持续成长。

### 阶段 4：基地建设系统完整版

目标：让基地建设成为主要策略层。

主要功能：

- 网格化建造。
- 建造预览、可建/不可建提示。
- 建筑旋转。
- 建筑拆除与部分资源返还。
- 建筑分类：核心、能源、生产、防御、存储、科研、特殊。
- 建筑占地大小。
- 建造范围限制。
- 建筑耐久与修复。
- 电力网络或供能范围。
- 防御塔 AI 核心数量限制。

验收标准：

- 玩家能规划基地布局。
- 防御塔不能无限堆叠。
- 建筑位置、供能、资源、占地都会影响决策。

### 阶段 5：机甲战斗与成长

目标：让玩家操作从“移动角色”升级成“机甲支援核心”。

主要功能：

- 机甲生命值、护盾、能量。
- 近战武器。
- 远程武器。
- 双武器槽切换。
- 冲刺/闪避。
- 武器弹药或能量消耗。
- 装备升级。
- 模组系统：伤害、攻速、射程、元素效果。
- 机甲死亡与复活惩罚。

验收标准：

- 玩家能主动处理防御塔覆盖不到的威胁。
- 武器配置会影响战斗手感和守家效率。
- 死亡会造成资源或防线压力。

### 阶段 6：科技树

目标：建立长线成长和内容解锁节奏。

主要功能：

- 科研建筑。
- 科技点或研究时间。
- 科技树 UI。
- 建筑解锁。
- 防御塔升级解锁。
- 机甲武器/模组解锁。
- 资源生产效率提升。
- 生物群系任务绑定科技前置。

验收标准：

- 玩家不能开局拥有所有高级能力。
- 科技研究能改变下一轮建设和战斗策略。
- 主线任务能推动关键科技解锁。

### 阶段 7：敌人生态与动态波次

目标：让敌潮从简单刷怪变成可调控的压力系统。

主要功能：

- 多敌人类型：小型群怪、重甲敌人、远程敌人、破墙敌人、精英敌人、Boss。
- 敌人抗性与弱点。
- 敌人刷新点。
- 多方向进攻。
- 动态目标管理器：根据基地强度、资源产能、科技进度调整波次。
- 特殊事件波次。
- 攻击预警 UI。

验收标准：

- 不同防御组合面对不同敌人有明显优劣。
- 波次压力随玩家发展提升。
- 玩家能提前获得进攻方向和倒计时信息。

### 阶段 8：关卡与地图结构

目标：做出战役推进的地图层。

主要功能：

- 主基地地图。
- 支线资源地图。
- 地图传送。
- 地图资源分布。
- 敌人巢穴。
- 可破坏障碍。
- 特殊地形。
- 地图探索雾或区域解锁。
- 生物群系规则：沙漠、沼泽、火山、冰原、酸蚀区等。

验收标准：

- 玩家需要离开安全区探索。
- 不同地图影响资源、敌人和建筑策略。
- 新地图能带来新资源和新科技。

### 阶段 9：战役模式

目标：实现完整主线流程。

主要功能：

- 任务系统。
- 任务追踪 UI。
- 主线目标。
- 支线目标。
- 剧情触发器。
- 新手引导。
- 关键建筑建造目标。
- 生物群系任务链。
- 最终裂隙/传送门建造与充能。
- 最终防御战。

验收标准：

- 玩家从开局到终局有明确目标链。
- 每个主线阶段都会引入新系统。
- 最终战需要综合资源、科技、基地和机甲能力。

### 阶段 10：生存模式

目标：提供无终点挑战。

主要功能：

- 生存模式开局配置。
- 无限波次。
- 平滑递增难度。
- 固定波次特殊敌人组合。
- 生存时长和波次数统计。
- 排行榜数据结构预留。

验收标准：

- 玩家可以反复挑战更高波次。
- 生存模式不依赖剧情任务也能完整运行。

### 阶段 11：沙盒模式

目标：提供自由调参和建造体验。

主要功能：

- 自定义资源倍率。
- 自定义敌人强度。
- 自定义波次间隔。
- 是否开启敌人。
- 是否开启死亡惩罚。
- 科技全解锁开关。
- 无限资源开关。

验收标准：

- 玩家能用沙盒快速测试基地设计。
- 沙盒参数不会污染战役和生存模式存档。

### 阶段 12：多人合作

目标：支持最多 4 人合作。

主要功能：

- 联机房间。
- 玩家同步。
- 资源池同步。
- 建筑同步。
- 敌人同步。
- 科技同步。
- 任务同步。
- 权限和冲突处理。
- 多人难度缩放。

验收标准：

- 多名玩家能在同一地图采集、建造、防守。
- 掉线和重连有基本处理。
- 多人模式不会破坏单人模式架构。

### 阶段 13：内容量产与表现打磨

目标：把系统原型变成完整产品。

主要功能：

- 完整建筑列表。
- 完整防御塔列表。
- 完整敌人列表。
- 完整武器与模组列表。
- 音效。
- 音乐。
- 粒子特效。
- 命中反馈。
- 屏幕震动。
- UI 动效。
- 设置菜单。
- 键位重绑定。
- 手柄支持。
- 存档/读档。
- 本地化。

验收标准：

- 系统、内容、表现形成统一体验。
- 玩家能长期推进，不只是在测试单个功能。

### 阶段 14：优化、测试与发布

目标：保证完整游戏稳定运行。

主要功能：

- 性能优化。
- 大量敌人寻路优化。
- 防御塔目标搜索优化。
- 存档兼容。
- 自动化冒烟测试。
- 平衡测试。
- 崩溃日志。
- 发布构建。

验收标准：

- 大规模敌潮下帧率稳定。
- 存档读档可靠。
- 关键流程无阻断 bug。
- 发布包可正常启动并完成一局。

## 7. 核心系统依赖顺序

必须优先实现：

1. 玩家控制。
2. 基地核心。
3. 资源。
4. 建造。
5. 敌人。
6. 防御塔。
7. 波次。
8. UI。

之后再实现：

1. 多资源。
2. 电力。
3. 生产链。
4. 科技树。
5. 机甲装备。
6. 关卡任务。
7. 生物群系。
8. 生存/沙盒。
9. 多人合作。

## 8. 第一阶段立即执行清单

下一步建议直接进入阶段 0 和阶段 1：

- 新建基础目录结构。
- 把当前玩家脚本迁移到 `scripts/player/`。
- 把当前主场景整理到 `scenes/main/`。
- 新建基地核心场景。
- 新建资源点场景。
- 新建敌人场景。
- 新建防御塔场景。
- 新建波次管理器。
- 新建 HUD。
- 做出“采集 -> 建塔 -> 防守 -> 胜负”的第一版。

## 9. 长期风险

- 系统量非常大，不能一开始同时做科技树、多人、复杂资源和完整战役。
- 大量敌人会带来寻路和性能压力，需要早期建立对象池和分批更新策略。
- 建造、电力、资源、波次之间高度耦合，需要用清晰事件和数据结构降低互相牵扯。
- 多人合作必须放在后期，但单人系统设计时要避免完全写死为单人。
- 当前背景图很大，后期需要切图、压缩或改为 TileMap/分块地图。

## 10. 最近版本目标

最近目标不是“做完全部功能”，而是做出第一条可以反复玩的核心循环。

版本名：`v0.1 基地防守原型`

包含内容：

- 玩家移动。
- 基地血量。
- 单资源采集。
- 基础建塔。
- 哨兵塔自动攻击。
- 敌人波次。
- 胜负结算。
- 基础 HUD。

完成 `v0.1` 后，再进入资源生产、电力、城墙、升级和科技树。
